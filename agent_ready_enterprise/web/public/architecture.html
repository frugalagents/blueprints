The Problem: Multi-Agent / MCP Server Sprawl

Traditional: Domain Agents + MCP Servers

  • One MCP server per domain (payroll, benefits, tax...)
  • Router agent decides which domain → routing errors
  • Cross-domain questions require multi-hop agent handoffs
  • Tool count grows → LLM selection accuracy collapses
  • Adding a domain = new server, new routes, new prompts, new tests
  • Context duplicated across agent boundaries
  • No shared semantic understanding

This: Single Agent + Semantic Knowledge Graph

  • One semantic layer covers ALL domains
  • Vector search replaces routing logic
  • Cross-domain via graph traversal — one pass
  • Agent sees only 5-10 relevant APIs per query
  • Adding a domain = drop specs, run 3 commands
  • Bounded context — never sees the full graph
  • Rich descriptions enable semantic matching

Design Principles

Semantic Layer = Retrieval

Rich descriptions embedded in vector store. Questions match capabilities through meaning. "Take-home pay" finds "net pay" without synonym mappings.

Graph = Structure Only

Dependency resolution, topological ordering, constraint validation, community detection. Never used for text matching — that's the semantic layer's job.

LLM Plans, Graph Validates

LLM generates execution plans from capability descriptions. Graph enforces structural correctness. Repair loop if validation fails.

Deterministic Execution

No generated Python. DAG executor runs typed API calls and operators (diff, attribute_delta, extract) in dependency order. Fully reproducible.

Zero Domain-Specific Code

No hardcoded keywords, templates, or routing rules. New domain = drop OpenAPI specs + 3 commands. Framework derives everything from API definitions.

Evidence-Grounded Answers

LLM synthesizes answers only from structured evidence claims. Every fact traces to a specific API call. Full audit trail from question to answer.

Adding a New Domain (Zero Code Changes)

# 1. Drop OpenAPI specs specs/sample_healthcare/*.yaml # 2. Build structural graph python -m api_kg.cli build --specs-dir ./specs/sample_healthcare # 3. Generate semantic descriptions (LLM-enriched) python -m api_kg.cli describe --specs-dir ./specs/sample_healthcare # 4. Build vector index + communities python -m api_kg.cli index # 5. Ask questions — works immediately python -m api_kg.cli ask "What drug interactions exist for patient P-1042?"
